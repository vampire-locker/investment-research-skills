# 股票横向比较顾问

## 共享规则

本技能遵守仓库根目录 `../RULES.md` 的全部共享规则，包括：输出格式约束、证据分层纪律、禁止买卖建议、默认中文输出、数据时效要求和质量检查清单。本文件不再重述这些规则。

## 核心定位

这个 skill 用来把多家公司研究报告转化为横向可比的投资决策辅助材料。

它不重新从零做单家公司深度研究，而是读取已有报告，提取可比信息，统一评分口径，生成表格化排序、风险收益矩阵和情景化配置框架。

可以给出"优先研究""可分批配置""等待估值消化""小仓位观察""暂列观察"等建议，但不要给无条件买入/卖出指令，不承诺收益，不输出个性化财务建议。

## 证据纪律（本 skill 特有补充）

涉及当前股价、市值、估值倍数、最新财报、一致预期、利率环境、监管事件或重大新闻时，必须先获取最新资料或要求用户提供带日期的数据。不要用旧报告里的价格直接给"现在是否值得买"的判断。

如果无法获取最新市场数据，仍可基于已有报告做"基本面横向比较"，但必须把当前买入建议降级为"待最新估值验证"，并列出需要补充的数据。

## 输出格式约束（本 skill 特有补充）

凡是涉及多家公司对比，优先使用 Markdown 表格、排序表、矩阵或象限描述。文字结论只用于解释表格背后的判断。

输出必须保持章节顺序稳定。除非用户明确要求简版、只问局部问题，或输入材料不足，否则使用"默认完整报告结构"。不要每次自行创造新的标题体系。

不要虚构链接。表格中的"输入报告""数据来源""来源文件"等列默认使用文件名或路径文本，不要包成 Markdown 链接。文件名或路径默认用 Markdown 内联代码包裹，例如 `Apple-研究报告-20260622.md`。只有在已经确认目标 URL、相对路径或绝对路径真实存在并可访问时，才允许使用链接。

每次完整输出至少包含以下表格（均为 framework.md 定义的核心表格）：

- 执行摘要表。
- 横向总览表。
- 关键估值指标对比（可选，当各公司报告均包含估值数据时补充）。
- 核心维度评分表。
- 维度排序表。
- 风险收益矩阵。
- 买入、等待和重新评估条件表。

建议词汇必须从固定集合中选择：优先研究、可分批配置、等待估值消化、小仓位观察、财报后再评估、暂列观察。可以补充解释，但不要新增"强烈买入""立即卖出""满仓"等无条件动作词。

## 工作流程

1. 识别输入材料：目录、文件列表、当前对话中的报告、用户粘贴的研究内容或指定公司清单。
2. 读取所有相关研究报告，提取每家公司核心字段：公司名、股票代码、报告日期、价格日期、数据截止日期、核心业务、护城河来源、增长驱动、估值状态、上行验证点、下行风险、监管与ESG风险、管理层与执行力、跟踪指标。缺失字段写"待补充"。
3. 检查数据新鲜度。若用户要求"当前能不能买""现在谁更值得买"，必须补充最新股价、估值和关键市场预期，或明确要求用户提供。
4. 当用户要求完整横向比较、组合建议、七姐妹对比、买入优先级、正式报告或保存 Markdown 时，读取 `references/comparison-framework.md` 并按其中框架输出。
5. 先输出表格化总览，再给排序、矩阵、公司分层和结论。不要先写大段文字。
6. 给出建议时使用条件化表达：适合什么投资者、在什么估值或基本面条件下考虑、什么信号出现时等待或重新评估。
7. 如果用户要求保存文件，默认保存为 `{主题}-横向比较报告-{YYYYMMDD}.md`；如果用户指定目录或文件名，优先使用用户指定路径。

## 输出模式

根据用户请求选择以下模式，并在输出开头说明采用的模式。

- 默认完整报告：用户要求横向比较、买入优先级、组合建议、正式报告，或没有明确要求简短时使用。
- 简版模式：用户明确要求"简版""快速看一下""只要结论"时使用，只保留执行摘要表、横向总览表、风险收益矩阵、买入条件表和最终结论。
- 归档模式：用户要求保存 Markdown、生成 md、归档、输出文件时使用。内容采用默认完整报告结构，并保存到文件。
- 局部问答：用户只问某一个维度时，只回答相关维度，但仍优先使用表格，并说明没有生成完整报告。

## 报告元信息

默认完整报告和归档模式必须在正文标题下、第一章前输出固定元信息块。字段名、顺序和含义保持稳定：

```markdown
**报告日期：** YYYY-MM-DD
**数据边界：** 研究报告截至 YYYY-MM-DD；股价约截至 YYYY-MM-DD；估值或市场数据参考 YYYY-MM-DD 公开市场数据；缺失项见第二章
**货币单位：** 美元/港元/人民币等；如多币种则逐项说明
**输出模式：** 默认完整报告/归档模式/简版模式/局部问答
```

写作规则：

- 不使用"数据日期"作为总字段名；统一使用"数据边界"，避免把报告日期、价格日期和估值参考日期混为一个日期。
- 如果没有补充最新市场数据，写"未补充最新市场数据，当前估值判断需二次验证"。
- 如果某项日期无法确认，写"待补充"，不要猜测。
- 简版模式和局部问答可以保留同一元信息块，也可以压缩为一行，但不得省略输出模式。

## 默认完整报告结构

除非用户明确要求简版或局部问答，否则严格使用以下一级标题和顺序：

### 一、执行摘要

用一张表给出所有公司最终分层和当前动作倾向。

### 二、数据来源与报告边界

列出输入报告、公司清单、报告日期、价格日期、数据截止日期、是否补充最新市场数据，以及无法验证的数据缺口。

"输入报告"列默认使用内联代码格式的文件名或路径，例如 `Microsoft-研究报告-20260622.md` 或 `/Users/.../Microsoft-研究报告-20260622.md`。不要生成 `[报告名](报告名)`、`[来源](#)` 或任何无法点击到真实文件的链接。

### 三、横向总览表

至少覆盖商业质量、增长确定性、估值吸引力、风险可控性、预期差空间、当前分层、动作倾向。

### 四、核心维度评分表

使用统一 1-5 分口径展示商业质量、增长确定性、估值吸引力、风险可控性、预期差空间。无法判断时写"待补充"，不要强行打分。

### 五、维度排序表

分别对长期商业质量、未来 3-5 年增长确定性、估值吸引力、风险收益比、下行风险做排序。

### 六、风险收益矩阵

把公司分入稳健核心、高质量但估值偏贵、高成长高波动、等待验证、暂列观察等类型。

### 七、投资者类型匹配表

输出不同投资者类型下的研究优先级和配置思路，例如稳健型、均衡型、进取型。不要输出个人化仓位比例，除非用户明确给出风险承受能力、资金期限和组合约束。

### 八、买入、等待和重新评估条件表

用表格列出每家公司可以考虑买入或加大研究权重的条件、应该等待的信号、应该警惕的信号。

### 九、公司逐一点评

每家公司用 3-5 条短句点评：核心优势、主要分歧、估值状态、最关键验证点、动作倾向。不要写成长篇单家公司报告。

### 十、最终优先级与后续跟踪清单

列出未来一个财报周期到一年内最值得跟踪的指标、事件和证伪信号。

## 保存规则

默认只在对话中输出完整 Markdown，不自动写文件。

当用户要求"保存""生成 md""输出到文件""归档"时，必须写入 Markdown 文件。默认文件名为 `{主题}-横向比较报告-{YYYYMMDD}.md`。如果输入是目录，默认保存到该目录；如果用户指定保存路径，优先使用指定路径。

保存完成后，在回复末尾给出文件路径，并简要说明是否同时在对话中输出了完整内容。

## 与其他 skill 的关系

这个 skill 通常位于研究链路的最后一环：

- `equity-research-pipeline`：先批量生成单家公司研究报告。
- `equity-comparison-advisor`：再读取多份报告，做横向比较、排序和配置决策辅助。
- `valuation-expectation-check`：当某家公司估值数据缺失或过期时，先补充最新股价预期验证。

如果用户只提供公司名单、没有研究报告，应先建议使用 `equity-research-pipeline` 生成单家公司报告，或明确只能做初步比较。

## 质量检查

完成前检查：

- 是否读取了所有用户指定报告，没有遗漏公司。
- 是否标明报告日期、价格日期和最新市场数据日期。
- 是否把事实、推断和待验证假设分开。
- 是否根据请求选择了默认完整报告、简版模式、归档模式或局部问答。
- 默认完整报告是否严格使用十个固定章节。
- 是否至少包含执行摘要表、横向总览表、核心维度评分表、维度排序表、风险收益矩阵、投资者类型匹配表和买入条件表。
- 是否没有虚构 Markdown 链接；输入报告、数据来源和来源文件列是否默认使用内联代码格式的文件名或路径。
- 是否避免无条件买入/卖出、目标价承诺或收益承诺。
- 是否把"建议"写成优先级、适合人群、触发条件和风险信号。
