## 一、背景

### 1.1 当前状态

investment-research-skills 没有经验沉淀机制。每次研究完成后，可复用的方法论洞察随对话结束而丢失。目前这类经验靠人工 commit 沉淀，例如：

- `fde4d8e` — "强化行业研究输入归因纪律"
- `50f0292` — "防止遗漏独立上市或改名公司的竞争玩家"

### 1.2 reverse-skill 的做法

reverse-skill 的 `field-journal/` 是三层架构：

```
field-journal/
├── _index.md           ← 按场景分类的经验索引
├── _template.md        ← 新建条目的标准模板
├── anonymization.md    ← 敏感信息脱敏规范
├── precedent-auth.md   ← 前置授权声明（每次任务前必须读）
├── precedent-reverse.md
├── precedent-pentest.md
├── seed-001 ~ 017      ← 种子知识（覆盖 17 个典型安全场景）
├── 2026-05-04-*.md     ← 实战日志（日期命名，来自真实任务）
├── 2026-05-05-*.md
└── ...
```

每次任务完成后 AI 自动回写经验到 journal，下次同类任务可参考。

## 二、投资研究场景的适配

### 2.1 三层结构设计

```
investment-research-skills/
└── field-journal/
    ├── _index.md                    ← 按研究阶段分类的经验索引
    ├── _template.md                 ← 新建条目的标准模板
    ├── seed-001-industry-cycle.md   ← 种子经验：行业周期位置判断
    ├── seed-002-moat-pressure.md    ← 种子经验：护城河压力测试常见遗漏
    ├── seed-003-valuation-traps.md  ← 种子经验：估值分析常见误区
    ├── seed-004-corp-structure.md   ← 种子经验：子公司分拆/上市主体识别
    ├── seed-005-cross-market.md     ← 种子经验：A/H 两地上市公司的处理
    ├── seed-006-cyclical-pe.md      ← 种子经验：周期股估值锚选择
    └── YYYY-MM-DD-{主题}.md        ← 实战日志（每次研究完成后自动生成）
```

### 2.2 模板 `_template.md`

```markdown
---
date: YYYY-MM-DD
company: 公司名
industry: 行业
skill_used: company-moat-research | sector-research | valuation-expectation-check | ...
tags: [方法论, 数据源, 边界情况, 常见错误]
---

# {一句话经验标题}

## 触发场景
什么情况下容易遇到这个问题。

## 问题描述
具体是什么问题（之前怎么做的，哪里出了问题）。

## 解决方法
应该怎么做（修改后的流程、新增的检查步骤、推荐的数据源）。

## 验证方式
如何确认这次没有再犯同样的错误。

## 关联
- [[另一条经验]]
- 来源 issue/commit: {ref}
```

### 2.3 种子经验示例

**seed-004-corp-structure.md：**

```markdown
---
date: 2025-06-25
tags: [边界情况, 行业研究, 公司研究]
---

# 行业竞争格局分析前必须检查公司结构变动

## 触发场景
- 画竞争格局表格前
- 列出行业主要玩家时
- 引用市场份额数据时

## 问题描述
分析行业时容易沿用旧的上市公司名称或漏掉新独立的实体。典型案例：
- SanDisk 从西部数据（WDC）分拆后独立上市，若仍按 WDC 旗下品牌处理会遗漏一个独立竞争玩家
- 中国公司在 A 股和港股两地上市，上市主体名称不同（如比亚迪股份 vs 比亚迪）
- 品牌名与上市主体名不一致（如抖音 vs 字节跳动未上市）

## 解决方法
在 sector-research 的工作流中增加"行业结构变动检查"步骤：
1. 搜索"行业 + 拆分/分拆/独立上市/合并/收购/更名/退市/新上市"
2. 对每个主要玩家确认其当前上市主体全称和交易所
3. 标注结构变动的时间点和影响

## 验证方式
检查竞争格局表格中是否每条公司名都可追溯到当前有效的上市主体。
```

### 2.4 实战日志生成规则

每次研究任务完成后，AI 自动执行：

1. 回顾本次研究中遇到的不顺利、需要额外搜索验证、或事后发现遗漏的点
2. 按模板格式生成一篇 journal 条目
3. 写入 `field-journal/YYYY-MM-DD-{主题}.md`
4. 更新 `_index.md`，加入新条目的索引行

### 2.5 索引 `_index.md` 结构

```markdown
# 经验索引

## 行业研究
- [2025-06-25 公司结构变动检查](2025-06-25-corp-structure-check.md) — 竞争格局前必查拆分/独立上市
- ...

## 护城河分析
- [2025-06-20 NAND 行业资本开支陷阱](...) — ...
- ...

## 估值验证
- ...
```

## 三、与 RULES.md 的集成

如果在后续版本实施本方案，RULES.md 中应新增一条：

```markdown
## 经验回写纪律

1. 每次研究任务完成后，回顾是否有可复用的方法论经验。
2. 如有，按 `field-journal/_template.md` 格式生成条目，写入 `field-journal/` 目录。
3. 更新 `field-journal/_index.md`。
4. 下个同类任务开始前，先查 `_index.md` 是否有相关经验可参考。
```

## 四、实施考量

| 维度 | 评估 |
| --- | --- |
| **价值** | 高。让每次研究都比上次更好，同类错误不再重复。目前靠人工 commit 沉淀，效率低且不结构化 |
| **实施复杂度** | 中。需要建立模板、种子经验、回写规则，但不需要修改已有 skill 的核心逻辑 |
| **与现有架构的冲突** | 无。纯增量，不影响任何现有 skill |
| **维护成本** | 低。模板固定后，新条目生成自动化 |
| **优先级** | 🟡 中。先完成共享规则层改造（整体安装），再做经验沉淀 |
