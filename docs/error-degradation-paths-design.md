## 一、背景

### 1.1 当前状态

investment-research-skills 各 skill 对数据缺失的处理是统一标"待补充"，缺乏分级降级策略。例如：

- 股价无法获取 → 标"待补充"，估值分析照常执行（但基础数据缺失，估值结论无意义）
- 财报数据缺失 → 护城河分析的财务验证章节标"待补充"
- 行业报告版本冲突 → 标"口径差异"

**问题："待补充"一视同仁，但不同数据缺失的影响完全不同。**

### 1.2 reverse-skill 的做法

reverse-skill 有明确的降级路径：

> - One way doesn't work, then change another — static→dynamic, Java→native .so
> - 工具两次自动安装失败 → 停止重试，输出手动安装步骤
> - 单条命令重复 ≥3 次 → 强制停止并评估
> - 接近 >30 次工具调用/子任务 → 上报用户

## 二、投资研究场景的分级框架

### 2.1 数据缺失的三个严重级别

| 级别 | 影响 | 处理策略 | 示例 |
| --- | --- | --- | --- |
| **L1 — 阻断级** | 核心结论不可靠，继续分析会产生误导 | 降级输出模式：从定量转为纯定性，明确告知用户 | 股价/市值无法获取时，估值分析不能给定量判断 |
| **L2 — 降级级** | 部分章节无法完整，但不影响整体分析框架 | 该章节标"因数据缺失跳过"，主体分析继续 | 一致预期缺失时，跳过"市场隐含增长反推"，但仍可做可比公司估值 |
| **L3 — 标注级** | 个别数据点缺失，不影响主线 | 原位标"待补充"或"数据暂缺"，不阻塞 | 个别季度的自由现金流数据缺失 |

### 2.2 各 skill 的降级路径

#### sector-research（行业研究）

| 缺失数据 | 级别 | 降级策略 |
| --- | --- | --- |
| 行业总规模/增速（最新年份） | L2 | 使用上一期数据，标注"基于 YYYY 年数据，最新数据待补充" |
| 市场份额数据 | L2 | 用定性描述替代定量排名（"市场集中度较高/分散"而非"CR5=XX%"） |
| 行业价格/产能数据 | L2 | 跳过周期位置中的定量判断，仍保留定性趋势分析 |
| 政策/监管信息 | L2 | 标注"本期未覆盖政策维度"，建议后续补充 |
| 所有核心行业数据都缺失 | L1 | 无法生成有效行业研究。输出已知信息片段，建议用户提供具体数据需求再启动 |

#### company-moat-research（护城河研究）

| 缺失数据 | 级别 | 降级策略 |
| --- | --- | --- |
| 分部收入/利润 | L2 | 使用公司整体数据，标注"分部数据待补充，以下分析基于公司整体" |
| 竞争对手的详细财务 | L3 | 定性比较替代定量，"竞争对手利润率大致在 [范围] 级别" |
| 最新财报数据 | L2 | 使用上一期财报，标注截止日期，提示"最新一期财报待补充" |
| 公司核心业务描述不清 | L1 | 无法进行护城河分析。回复用户需要哪些基本材料才能启动 |

#### valuation-expectation-check（估值验证）

| 缺失数据 | 级别 | 降级策略 |
| --- | --- | --- |
| 当前股价 | **L1** | 自动降级为纯定性讨论，不给定量估值判断。输出标题改为"估值框架讨论"而非"股价预期验证" |
| 市值/企业价值 | L1 | 同上 |
| 一致预期 | L2 | 跳过"市场隐含增长反推"，仅做可比估值和历史估值区间对照 |
| 可比公司估值倍数 | L2 | 只用历史估值区间对照，标注"缺少可比公司数据" |
| 自由现金流 | L2 | 改用 PE 或 EV/EBITDA 作为主要估值锚，标注切换原因 |
| 汇率（非本币） | L3 | 使用最近可用汇率，标注日期 |

#### equity-comparison-advisor（横向比较）

| 缺失数据 | 级别 | 降级策略 |
| --- | --- | --- |
| 某公司完整报告缺失 | L2 | 该公司的所有维度标"待补充"，不参与评分和排序，仅列在"数据不足"分区 |
| 多家公司估值数据过期 | L2 | 估值相关维度（估值吸引力）全部降级，比较报告增加提示："估值维度基于 [日期] 数据，未补充最新市场数据" |
| 仅剩 1 家有完整数据 | L1 | 无法做横向比较。回复用户建议先补全其他公司报告 |
| 评级口径不一致 | L2 | 统一为 1-5 分，标注"不同来源的评级标准可能不完全可比" |

### 2.3 搜索降级路径

| 情况 | 策略 |
| --- | --- |
| 首次搜索未找到数据 | 切换搜索词（中文↔英文、公司全称↔代码） |
| 第二次搜索仍未找到 | 切换数据来源（Yahoo→Google Finance、东方财富→巨潮资讯网） |
| 第三次搜索仍未找到 | 停止搜索。根据缺失数据所在级别执行对应降级策略，在输出中标注"经多次搜索未获取 [数据名]，已按 L{级别} 降级处理" |
| 同一数据点跨 3 个来源获取到矛盾数据 | 标注所有来源和各自数据，不做调和，提示用户"存在口径差异，需自行确认" |

## 三、与 RULES.md 的集成

如果在后续版本实施本方案，在 RULES.md 中增加：

```markdown
## 错误降级路径

### 数据获取失败时的行为

1. 对缺失数据按严重程度分级（L1 阻断/L2 降级/L3 标注），执行对应策略。
2. 同一数据点最多尝试 3 次（不同搜索词或不同来源）。3 次仍未获取则停止，执行降级。
3. 不同来源数据矛盾时，标注所有来源和各自数据，不做调和。
4. L1 级别数据缺失导致核心结论不可靠时，必须向用户说明：无法完成该项分析的哪一部分、需要补充什么数据才能继续。

各 skill 的具体降级路径定义见各 skill 的 SKILL.md。

### 单次分析的复杂度控制

1. 同一问题工具调用接近 30 次时，暂停并向用户汇报当前进展、确认是否继续。
2. 同一搜索关键词重复搜索 ≥3 次时，停止并切换搜索方式。
```

## 四、实施考量

| 维度 | 评估 |
| --- | --- |
| **价值** | 中。让 agent 在数据不完整时的表现更可控、输出更诚实。当前"统一待补充"可能误导用户以为数据缺口不重要 |
| **实施复杂度** | 中。需要为每个 skill 定义降级路径，但不需要修改 skill 的核心方法论 |
| **与 SKILL.md 的关系** | 降级路径应该写在各自 SKILL.md 中（因为是 skill 特有的），通用原则写在 RULES.md 中 |
| **优先级** | 🟢 低。当前"统一待补充"模式没有出过重大问题。建议等共享规则层完成后，在实测中发现具体降级行为不足时再实施 |
